## How webhooks work

When a file finishes processing, a payment is confirmed, or a student email account is provisioned, Nexa sends a signed `POST` request to a URL you register in the dashboard. The SDK:

1. Verifies the HMAC-SHA256 signature using your `NEXA_WEBHOOK_SECRET`
2. Rejects events older than 5 minutes (replay protection)
3. Parses the payload into a typed `WebhookEvent`
4. Calls your `onFileComplete` or `onPaymentComplete` callback

All of this happens automatically inside the catch-all route handler — you only implement the callbacks.

---

## Setup checklist

**1. Register your webhook URLs** in the Nexa dashboard:

| Event | URL |
|-------|-----|
| `file.completed` | `https://your-domain.com/api/nexa/webhook` |
| `payment.completed` | `https://your-domain.com/api/nexa/payments/forward` |
| `email.bulk_completed` | `https://your-domain.com/api/nexa/email-provisioning/forward` |

**2. Confirm `NEXA_WEBHOOK_SECRET` is set** in your environment:

```bash
NEXA_WEBHOOK_SECRET=whsec_...
```

**3. Implement your callbacks** in `lib/nexa.ts`:

```ts
export const nexa = createNexa({
  apiKey: process.env.NEXA_API_KEY!,
  webhookSecret: process.env.NEXA_WEBHOOK_SECRET!,
  getUser: async () => { /* ... */ },

  onFileComplete: async (event) => {
    // called when file.completed arrives
    await db.fileResults.upsert({ fileId: event.fileId, userId: event.userId });
  },

  onPaymentComplete: async (event) => {
    // called when payment.completed arrives
    await db.payments.upsert({ reference: event.reference, status: event.status });
  },

  onEmailBulkCompleted: async (event) => {
    // called when email.bulk_completed arrives (bulk job finished)
    // event contains counts only — call nexa.email.getJobStatus(event.jobId) for per-student results
    console.log(`Bulk job ${event.jobId}: ${event.successCount}/${event.totalStudents} created`);
    if (event.failedCount > 0) {
      const details = await nexa.email.getJobStatus(event.jobId);
      const failed = details.results?.filter(r => r.status === "failed");
      await notifyAdmin(failed);
    }
  },
});
```

## Manual verification

If you need to verify a webhook in a custom route outside the SDK:

```ts
import { nexa } from "@/lib/nexa";

// app/api/custom-webhook/route.ts
export async function POST(request: Request) {
  const event = await nexa.webhooks.verify(request);
  // throws NexaError if signature is invalid or event is too old

  if (event.type === "file.completed") {
    await handleFileComplete(event);
  }

  return Response.json({ received: true });
}
```

You can also adjust the replay protection window:

```ts
const event = await nexa.webhooks.verify(request, { maxAgeSeconds: 600 });
```

## `WebhookEvent` types

```ts
type WebhookEvent =
  | WebhookFileCompleteEvent
  | WebhookPaymentEvent
  | WebhookEmailBulkCompletedEvent;
```

### `WebhookFileCompleteEvent`

```ts
type WebhookFileCompleteEvent = {
  type: "file.completed";
  fileId: string;
  userId: string;
  tenantId: string;
  status: "success" | "failed";
  records?: StudentRecord[];
  error?: string;
};
```

### `WebhookPaymentEvent`

```ts
type WebhookPaymentEvent = {
  type: "payment.completed";
  reference: string;
  tenantId: string;
  customerEmail: string;
  amount: number;
  amountPaid: number;
  status: "success" | "failed";
  fees: number;
  platformFee: number;
  netAmount: number;
  paidAt: string | null;
  channel: string;
  failureReason: string | null;
  createdAt: string;
  updatedAt: string;
};
```

### `WebhookEmailBulkCompletedEvent`

Fired once when a bulk provisioning job reaches `completed` or `failed`. The payload is lean — counts only. Call `nexa.email.getJobStatus(jobId)` to retrieve the full per-student breakdown (individual emails, error details).

```ts
type WebhookEmailBulkCompletedEvent = {
  event: "email.bulk_completed";
  tenantId: string;
  jobId: string;
  status: "completed" | "failed";
  totalStudents: number;
  successCount: number;
  failedCount: number;
  completedAt: number;  // Unix ms
};
```

<Callout type="info">
  **Why no per-student data in the webhook?**
  Pushing individual account details for each student in a 500-student job would make the webhook body extremely large and expensive to receive and process. Instead, the webhook tells you the job is done — you pull the full results only when you need them via `nexa.email.getJobStatus(jobId)`.
</Callout>

## Convex integration

`@nexa-ed/convex` provides pre-built handlers that sync events straight into your Convex backend:

```ts
import { createFileCompleteHandler, createPaymentCompleteHandler } from "@nexa-ed/convex/handlers";
import { ConvexHttpClient } from "convex/browser";
import { api } from "../convex/_generated/api";

const convex = new ConvexHttpClient(process.env.NEXT_PUBLIC_CONVEX_URL!);

export const nexa = createNexa({
  // ...
  onFileComplete: createFileCompleteHandler(convex, api),
  onPaymentComplete: createPaymentCompleteHandler(convex, api),
});
```

This writes to the `nexaFileResults`, `paymentTransactions`, and `studentEmails` tables defined by `@nexa-ed/convex/schema`.

For email bulk completion, use `createStudentEmailHandler` alongside an `onEmailBulkCompleted` callback:

```ts
import {
  createFileCompleteHandler,
  createPaymentCompleteHandler,
} from "@nexa-ed/convex/handlers";

export const nexa = createNexa({
  // ...
  onFileComplete:    createFileCompleteHandler(convex, api),
  onPaymentComplete: createPaymentCompleteHandler(convex, api),

  onEmailBulkCompleted: async (event) => {
    // Webhook fires once per job. Pull full results from the API.
    const details = await nexa.email.getJobStatus(event.jobId);
    // Sync each provisioned account to your Convex DB, send notifications, etc.
  },
});
```

## Testing webhooks locally

Use [ngrok](https://ngrok.com) or [Cloudflare Tunnel](https://developers.cloudflare.com/cloudflare-one/connections/connect-networks/) to expose your local server:

```bash
ngrok http 3000
# → Forwarding https://abc123.ngrok.io → localhost:3000
```

Register `https://abc123.ngrok.io/api/nexa/webhook` in the Nexa dashboard temporarily, then trigger a test event from the dashboard to verify your handler is working.
