## Bug Fixes

### All packages — `package.json` exports map was broken

**Affected:** `@nexa-ed/sdk` · `@nexa-ed/next` · `@nexa-ed/react` · `@nexa-ed/convex` · `@nexa-ed/node`

Two compounding bugs made every SDK package unresolvable when installed from npm:

**Bug 1 — Regular exports pointed to TypeScript source.**
All packages had `exports`, `main`, and `module` pointing to `./src/index.ts`. Since `files` only includes `dist/`, the `src/` directory is never in the published tarball. Any scaffolded app received a package referencing a file that does not exist on disk.

**Bug 2 — `publishConfig` referenced `.mjs` but tsup outputs `.js`.**
Because `"type": "module"` is declared in each package, tsup writes the ESM bundle as `dist/index.js` — not `dist/index.mjs`. The `publishConfig` pointed to `./dist/index.mjs`, a file that was never generated, so Next.js webpack threw:
```
Module not found: Can't resolve '@nexa-ed/react'
```

**Bug 3 — `types` condition was ordered after `import`/`require`.**
TypeScript requires the `types` condition to come first in the exports map so it can resolve type declarations. With `types` listed last, TypeScript silently failed to find declarations for workspace packages, which caused downstream DTS builds (e.g. `@nexa-ed/node` generating types for `express.ts`) to error with `Cannot find module '@nexa-ed/sdk'` — even though the package was present.

**Fix applied to all 5 packages:**

```diff
  "exports": {
    ".": {
-     "default": "./src/index.ts"
+     "types":   "./dist/index.d.ts",
+     "import":  "./dist/index.js",
+     "require": "./dist/index.cjs"
    }
  },
- "main":  "./src/index.ts",
- "module": "./src/index.ts",
- "types": "./src/index.ts"
+ "main":  "./dist/index.cjs",
+ "module": "./dist/index.js",
+ "types": "./dist/index.d.ts"
```

`publishConfig` was updated in the same way (`.mjs` → `.js`).

### Publish workflow — pre-publish typecheck gate

A `Type-check all SDK packages` step now runs after the build and before changesets publishes. A broken DTS will fail the workflow immediately with a clear TypeScript error rather than blowing up inside `changeset publish` with a cryptic `ELIFECYCLE` exit code.

## What Changed

### `create-nexaed-app` — CLI commands and CI support

#### Non-interactive mode

All prompts can now be bypassed with flags. This lets `create-nexaed-app` run in CI pipelines, Dockerfiles, and shell scripts without a TTY.

```bash
npx create-nexaed-app my-school \
  --auth clerk \
  --features file-processing,payments \
  --email-tier tier-1-nexa \
  --api-key nxa_live_... \
  --pm pnpm
```

**Supported flags:**

| Flag | Description |
|------|-------------|
| `--auth <provider>` | `clerk` · `nextauth` · `none` |
| `--features <list>` | Comma-separated: `file-processing`, `payments`, `student-email` |
| `--email-tier <tier>` | `tier-1-nexa` · `tier-2-stalwart` · `tier-3-google` |
| `--email-domain <domain>` | Your school domain (required for tier 2/3) |
| `--api-key <key>` | Pre-fill the Nexa API key in `.env.local` |
| `--pm <manager>` | `pnpm` · `npm` · `yarn` · `bun` |
| `--no-install` | Skip dependency installation |

When stdin is not a TTY (detected automatically), the non-interactive fast path activates and skips clack entirely — no manual flag passing required in most CI environments.

---

#### Command router

The CLI now dispatches to subcommands. `init` (scaffold) remains the default when no subcommand is given.

```bash
npx create-nexaed-app                     # init (default)
npx create-nexaed-app init my-school      # explicit init
npx create-nexaed-app doctor              # validate existing app
npx create-nexaed-app upgrade             # upgrade @nexa-ed/* deps
npx create-nexaed-app telemetry on        # enable telemetry
npx create-nexaed-app telemetry off       # disable telemetry
npx create-nexaed-app help                # print usage
```

---

#### `doctor` command

Validates an existing Nexa-powered school app against 5 checks:

```bash
cd my-school
npx create-nexaed-app doctor
```

| Check | What it verifies |
|-------|-----------------|
| **Env vars** | `NEXA_API_KEY` and `NEXA_WEBHOOK_SECRET` are present in `.env.local` |
| **Packages** | All required `@nexa-ed/*` packages are listed in `package.json` |
| **Route** | The catch-all `app/api/nexa/[...nexaed]/route.ts` exists |
| **SDK version** | Your installed version vs the latest on npm |
| **API key** | Makes a live request to the Nexa platform to confirm the key is valid |

Each check prints a pass/fail indicator. Run `doctor` after upgrading or when debugging a broken integration.

---

#### `upgrade` command

Updates all `@nexa-ed/*` dependencies in the current project to `@latest` using whichever package manager the project uses.

```bash
cd my-school
npx create-nexaed-app upgrade
```

Equivalent to running:

```bash
pnpm add @nexa-ed/sdk@latest @nexa-ed/react@latest @nexa-ed/next@latest ...
```

The package manager is detected from `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, or `bun.lockb`.

---

#### Opt-in telemetry

The CLI asks once on the first interactive run whether you want to share anonymous usage data. The preference is saved to `~/.nexaed/config.json` and never asked again.

- **Skipped automatically in CI** (non-TTY environments)
- **Fire-and-forget** — never delays the CLI or throws on failure
- **What is collected:** command name, node version, OS platform — no project contents or API keys

Toggle at any time:

```bash
npx create-nexaed-app telemetry on
npx create-nexaed-app telemetry off
```

---

#### Update notifier

On every run, the CLI checks npm in the background for a newer version of `create-nexaed-app`. If one is found, a one-line notice prints after the outro:

```
  ╭─────────────────────────────────────────╮
  │  Update available: 0.2.0-beta.3 → 0.2.0-beta.4    │
  │  Run: npx create-nexaed-app@latest      │
  ╰─────────────────────────────────────────╯
```

The check is background-only and never blocks the main flow.

## Upgrading from beta.3

**If you scaffolded an app with `0.2.0-beta.3`**, update all `@nexa-ed/*` packages immediately — they were broken when installed from npm:

```bash
pnpm add @nexa-ed/sdk@0.2.0-beta.4 \
         @nexa-ed/next@0.2.0-beta.4 \
         @nexa-ed/react@0.2.0-beta.4 \
         @nexa-ed/convex@0.2.0-beta.4 \
         @nexa-ed/node@0.2.0-beta.4
```

Or use the new `upgrade` command from inside your project:

```bash
npx create-nexaed-app upgrade
```

All import paths and API signatures are unchanged from `0.2.0-beta.3`.
